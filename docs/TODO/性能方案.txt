 待尝试方案："Shared Memory (MMF) + Fury" 方案
 
 
 grpc方案
 
 
 
 针对 100GB 级别的超大型项目，最佳方案是“句柄化”异步架构。核心思路是：协议（Control Plane）与数据（Data Plane）分离。
放弃将数据“塞进”协议的思路，改为通过协议传递“访问门票”。
最佳技术栈：AIDL + SharedMemory + FlatBuffers1. 控制层（Control Plane）：AIDL + FlatBuffers

* 为什么用 FlatBuffers：它支持 Zero-copy。数据在内存中无需反序列化即可读取。相比 lsp4j 的 JSON 解析，性能提升数十倍且内存占用极低。
* AIDL 的职责：仅负责指令调度（如 sync()、build()）和传递 Memory Address 或 File Descriptor。

2. 数据层（Data Plane）：SharedMemory + PFD

* 针对配置元数据（如依赖树）：使用 android.os.SharedMemory。
* 服务端在内存中生成一次 FlatBuffers 字节流。
   * 通过 AIDL 将 SharedMemory 对象传给 IDE。
   * IDE 进程直接映射（mmap）该内存。100GB 的项目元数据，传输耗时几乎为 0。
* 针对源码/大型文件：使用 ParcelFileDescriptor (PFD)。
* IDE 永远不请求“文件内容字符串”。
   * IDE 请求 getFile(path)，服务端返回一个 PFD。
   * IDE 利用 FileChannel 直接从内核读取数据，绕过所有 RPC 缓冲区。

3. 架构设计图

   1. Registry（索引层）：服务端启动时，将项目所有文件的路径、偏移量、依赖关系构建成一个 FlatBuffers 索引表。
   2. Mapping（映射层）：将该索引表放入 SharedMemory。
   3. Client（IDE端）：
   * 通过 AIDL 获取索引表的内存句柄。
      * 需要看代码时，根据索引表中的路径，向服务端索要 PFD。
      * 需要搜索时，直接在 mmap 的内存中进行位偏移查找，不产生 GC。
   
4. 关键代码语义示例 (AIDL)

import android.os.SharedMemory;import android.os.ParcelFileDescriptor;
interface IGradleBuildService {
    // 获取整个项目结构图的共享内存句柄 (FlatBuffers 格式)
    SharedMemory getProjectStructureIndex();

    // 针对单个大文件，直接获取文件句柄，实现零拷贝读取
    ParcelFileDescriptor getFileDescriptor(String relativePath);

    // 异步执行构建，仅通过回调传递状态码和进度，不传大数据
    void executeTask(String taskName, IBuildCallback callback);
}

方案优势

* 极致内存效率：无论项目多大，IDE 和 Gradle 进程共享同一份内存索引，不会出现两份 100GB 的缓存。
* 彻底解决 Binder 限制：Binder 只负责传递 4 字节的句柄，真正的数据流走内核 sendfile 或 mmap。
* 类型安全：AIDL 提供了强类型校验，比 JSON-RPC 更难出错。

建议步骤：先将 lsp4j 中的文件获取接口重构为 PFD (ParcelFileDescriptor) 模式，这能立即解决 1MB 传输限制并极大提升大型文本的加载速度。
